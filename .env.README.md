# Environment Configuration Guide

This directory contains environment variable templates for different deployment scenarios.

## Files Overview

### `.env.example` (Template - Source of Truth)
- **Purpose:** Documentation of ALL available environment variables
- **When to use:** Reference for which variables exist and what they do
- **Never commit:** Actual secrets - this is a template only
- **Audience:** Developers and DevOps engineers

### `.env.dev` (Local Development)
- **Purpose:** Pre-configured for local development with Docker services
- **When to use:** `cp .env.dev .env` for local development
- **Services:** Uses localhost for all services (PostgreSQL, Redis, Memgraph, etc.)
- **API Keys:** Minimal setup - only requires GEMINI_API_KEY
- **Tracing:** Disabled by default (verbose in development)
- **Example:**
  ```bash
  cp .env.dev .env
  # Fill in GEMINI_API_KEY
  docker-compose up -d  # Start local services
  bun run dev
  ```

### `.env.prod` (Railway Production)
- **Purpose:** Railway deployment configuration
- **When to use:** Production deployments on Railway
- **Services:** External managed services (Supabase, Railway PostgreSQL, Upstash Redis, R2, etc.)
- **API Keys:** All required for production features
- **Deployment:** Variables set in Railway Dashboard (NOT in git)
- **Monitoring:** Sentry and LangSmith enabled
- **Example:**
  ```bash
  # In Railway Dashboard:
  # 1. Attach PostgreSQL and Redis services
  # 2. Set environment variables from .env.prod
  # 3. Railway auto-configures DATABASE_URL and REDIS_URL
  ```

## Quick Start

### Local Development
```bash
# 1. Copy development template
cp .env.dev .env

# 2. Set only GEMINI_API_KEY
GEMINI_API_KEY=your-api-key

# 3. Start local services (optional)
docker-compose up -d

# 4. Run development server
bun run dev
```

### Production Deployment (Railway)
```bash
# 1. Push code to GitHub
git push origin main

# 2. In Railway Dashboard:
#    - Create new project from GitHub
#    - Attach PostgreSQL service
#    - Attach Redis service
#    - Set environment variables from .env.prod

# 3. Configure variables in Railway:
#    - GEMINI_API_KEY
#    - SUPABASE_* (auth)
#    - JWT_SECRET (generate new)
#    - S3_* (for file uploads)
#    - Optional: LANGSMITH_*, SENTRY_DSN
```

## Environment-Specific Behavior

| Variable | Dev | Prod |
|----------|-----|------|
| NODE_ENV | `development` | `production` |
| LOG_LEVEL | `debug` | `info` |
| Database | localhost:5432 | Railway PostgreSQL |
| Redis | localhost:6379 | Railway Redis |
| LANGSMITH_TRACING | `false` | `true` |
| Monitoring | None | Sentry + LangSmith |

## Security Notes

⚠️ **CRITICAL:**
- Never commit `.env` files with actual secrets
- Use `.env.example` for documentation only
- Railway Dashboard for secrets storage (not this repo)
- All API keys should be treated as sensitive

✅ **Best Practices:**
- `.env.example` - Commit to git (no secrets)
- `.env.dev` - Commit to git (safe for dev, no production keys)
- `.env.prod` - Set in Railway Dashboard (NOT in git)
- `.env` - Add to `.gitignore` (local only)

## Troubleshooting

### "GEMINI_API_KEY is not configured"
```bash
# Check your .env file has GEMINI_API_KEY set
grep GEMINI_API_KEY .env

# If empty, set it:
echo "GEMINI_API_KEY=your-key" >> .env
```

### "DATABASE_URL is not configured"
```bash
# For development: use .env.dev
cp .env.dev .env
bun run dev

# For production: Railway auto-configures this
# If missing, attach PostgreSQL service in Railway Dashboard
```

### "Cannot connect to Redis"
```bash
# For development: start Redis container
docker run -d -p 6379:6379 redis:latest

# For production: Railway auto-configures this
# If missing, attach Redis service in Railway Dashboard
```

## Updating Environment Variables

When adding new environment variables:

1. Add to [.env.example](.env.example) with documentation
2. Add defaults or empty values to appropriate template (.env.dev/.env.prod)
3. Update [.env.audit.md](.env.audit.md) if significant
4. Commit and push
5. For production: add to Railway Dashboard Variables

Example:
```bash
# In .env.example:
MY_NEW_VAR=                             # Description and source

# In .env.dev:
MY_NEW_VAR=local-value

# In .env.prod:
MY_NEW_VAR=                             # Set in Railway Dashboard
```
